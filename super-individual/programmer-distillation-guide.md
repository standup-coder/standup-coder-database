# 如何蒸馏一个程序员：技能萃取与知识留存全指南

## 1. 概述

"蒸馏"原本是化学中的提纯工艺，在AI领域指将大模型的知识迁移到小模型。对于程序员而言，**蒸馏 = 将个人多年积累的隐性经验、肌肉记忆和解决问题能力，系统地转化为可记录、可传递、可复用的显性知识资产**。

每一个程序员都是一座知识金矿——你的排障经验、架构直觉、踩坑记忆、代码风格、调试技巧都极具价值，但它们往往以隐性知识的形式存在，难以直接传递。本指南提供一套完整的程序员蒸馏方法论，帮助你把"会做"变成"能说能教能传"，让经验不再随离职而消失。

**核心目标**：

```
蒸馏前：经验在脑子里 → 只能自己用 → 离职即归零
蒸馏后：经验变成文档/工具/课程 → 团队可复用 → 持续创造价值
```

**适用场景**：
- 技术Leader萃取团队核心技能，降低人员流失风险
- 资深工程师系统化梳理个人知识体系
- 超级个体将技能转化为可变现的知识产品
- 企业建立技术知识库和培训体系
- 程序员为AI辅助编程准备高质量上下文

## 2. 蒸馏框架：程序员知识分层模型

### 2.1 四层知识金字塔

```
              ┌───────────┐
              │  元认知层   │  思维模型/决策直觉/审美判断
              │  (最难萃取) │  "为什么这样做"
              ├───────────┤
              │  方法论层   │  架构方法/调试策略/设计原则
              │  (需深度对话)│  "怎么系统性地做"
              ├───────────┤
              │  技能层     │  编程/调试/部署/运维具体操作
              │  (可观察萃取)│  "怎么做"
              ├───────────┤
              │  知识层     │  语言/框架/工具/API/协议
              │  (最容易萃取)│  "是什么"
              └───────────┘
```

| 层级 | 内容特征 | 萃取难度 | 萃取方法 | 产出形式 |
|------|---------|---------|---------|---------|
| 知识层 | 事实性信息，可直接记录 | 低 | 文档化、清单化 | 速查手册、Wiki、Anki卡片 |
| 技能层 | 操作性能力，需要演示和练习 | 中 | 录屏、结对编程、SOP | 视频教程、操作手册、脚本 |
| 方法论层 | 经验性判断，需要深度反思 | 高 | 案例复盘、深度访谈、思维导图 | 决策树、检查清单、框架文档 |
| 元认知层 | 直觉和审美，难以言传 | 极高 | 长期跟踪、情境模拟 | 原则清单、思维模型卡片 |

## 3. 萃取方法论

### 3.1 七种核心萃取技术

| 技术 | 适用层级 | 操作方式 | 所需时间 | 产出 |
|------|---------|---------|---------|------|
| **文档挖掘** | 知识层 | 梳理已有文档、代码注释、Commit记录 | 2-4小时 | 知识地图、术语表 |
| **屏幕录制** | 技能层 | 录制完成特定任务的全过程 | 1-3小时/任务 | 视频教程、操作SOP |
| **结对复盘** | 技能层+方法论层 | 边做边讲，搭档记录关键决策点 | 2-4小时/场景 | 决策日志、检查清单 |
| **案例深挖** | 方法论层 | 选取3-5个代表性案例深度复盘 | 2-3小时/案例 | 案例库、模式总结 |
| **关键事件访谈** | 方法论层+元认知层 | 围绕关键决策节点的深度对话 | 1-2小时/事件 | 决策树、原则清单 |
| **思维流追踪** | 元认知层 | 实时记录思考过程和决策理由 | 持续1-2周 | 思维模型文档 |
| **代码考古** | 全层级 | 对历史代码进行"考古式"分析 | 4-8小时/模块 | 架构演进文档、设计原则 |

### 3.2 文档挖掘：从已有资产中提炼

```
信息源梳理清单：

├── □ README.md —— 项目定位、技术选型、架构概述
├── □ docs/ 目录 —— 已有技术文档
├── □ Git log —— 提交历史中的决策痕迹
├── □ PR/MR 描述 —— 功能和设计决策
├── □ Code Review 评论 —— 技术讨论和权衡
├── □ .editorconfig / ESLint / Prettier —— 代码规范
├── □ Dockerfile / docker-compose.yml —— 部署知识
├── □ CI/CD 配置文件 —— 流水线知识
└── □ 测试文件 —— 质量保障策略

协作平台：
├── □ Confluence/Wiki —— 技术文档和决策记录
├── □ JIRA/Issue 系统 —— 需求理解和问题历史
├── □ Slack/钉钉 历史消息 —— 隐性知识碎片
├── □ 技术方案文档 —— 架构设计思路
└── □ 事故报告(Postmortem) —— 故障处理经验

├── □ 笔记和知识库 —— 个人积累
├── □ 技术博客 —— 已输出的内容

```

### 3.3 屏幕录制：捕获操作技能

```
录制SOP：

录制前准备：
├── 明确录制目标（一个具体任务）
├── 准备演示环境（干净的示例项目）
├── 关闭无关窗口和通知
├── 准备大纲（关键步骤清单）
└── 预演一遍（估算时间）

录制时注意：
├── 开启屏幕录制 + 麦克风
├── 边操作边讲解（"为什么这样做"比"做了什么"更重要）
├── 遇到错误不要剪掉（排障过程最有价值）
├── 标注关键决策点
└── 控制时长（单段15-30分钟为佳）

录制后处理：
├── 添加章节标记和目录
├── 补充文字版操作步骤
├── 标注关键时间点
├── 整理为SOP文档
└── 存入知识库
```

**推荐录制场景**：
- 从零搭建一个项目的完整过程
- 复杂Bug的排查和修复过程
- 线上故障的应急处理过程
- 代码审查的思路和要点
- 性能问题的定位和优化
- 新环境的配置和部署

### 3.4 案例深挖：STAR-Plus复盘法

```
STAR-Plus 案例复盘模板：

S (Situation) 背景：
├── 项目/系统是什么？
├── 技术栈和团队组成？
├── 业务背景和约束条件？
└── 当时面临的核心挑战？

T (Task) 任务：
├── 你的具体职责是什么？
├── 需要解决什么问题？
├── 成功标准是什么？
└── 时间和资源约束？

A (Action) 行动：
├── 你做了什么？（具体步骤）
├── 为什么选择这个方案？（决策逻辑）
├── 还考虑过哪些替代方案？（权衡过程）
├── 遇到了什么意外？怎么处理的？
└── 和谁协作了？如何协调的？

R (Result) 结果：
├── 最终结果如何？（量化数据）
├── 对业务/团队的影响？
├── 各方反馈如何？
└── 有什么可衡量的改进？

+ (Plus) 反思萃取：
├── 如果重来会怎么做？（改进空间）
├── 这个案例体现了什么通用原则？（抽象提炼）
├── 哪些决策是关键的？为什么？
├── 有什么可以复用的模式/方法/工具？
└── 有什么教训值得记住？（避坑指南）
```

### 3.5 关键事件访谈：深度萃取方法论

```
访谈问题框架（面向被萃取者）：

场景还原：
├── "在这个项目中，最让你头疼的技术决策是什么？"
├── "你是怎么做出最终选择的？考虑了哪些因素？"
├── "当时有没有哪些信息是你事后觉得应该更重视的？"
└── "如果让你重新来过，你会做哪些不同的选择？"

经验萃取：
├── "在你处理过的故障中，哪一个最有教育意义？"
├── "你判断一个技术方案好坏的标准是什么？"
├── "你面试候选人的时候最看重什么？为什么？"
├── "你学新技术的方法是什么？能具体描述一次吗？"
└── "你觉得这个领域最核心的3个原则是什么？"

直觉捕捉：
├── "你看到什么样的代码会立刻觉得'有问题'？"
├── "你做Code Review时第一个看的是什么？"
├── "你觉得什么样的架构设计是'优雅'的？"
├── "面对一个全新的技术问题，你的第一反应是什么？"
└── "你有没有遇到过'直觉告诉我不对，但说不清为什么'的情况？"

隐性知识外化：
├── "你能举一个你靠经验避开了大坑的例子吗？"
├── "你做技术选型时的checklist是什么？"
├── "什么情况下你会选择'不优化'？"
├── "你怎么判断一个技术债务该不该还？"
└── "你觉得'高级工程师'和'中级工程师'最大的区别是什么？"
```

### 3.6 代码考古：从代码中提取设计智慧

```
代码考古方法论：

第一步：选择目标模块
├── 选择核心模块（业务价值高、复杂度高）
├── 选择频繁修改的模块（变化热点）
└── 选择跨团队依赖的模块（接口关键）

第二步：历史追溯
├── git log --follow 追踪文件修改历史
├── git blame 分析每行代码的来源
├── 分析commit message中的决策线索
├── 追踪PR/MR中的设计讨论
└── 绘制模块变更时间线

第三步：模式识别
├── 设计模式使用情况（哪些、为什么、效果如何）
├── 架构演进轨迹（从V1到现在的变化）
├── 重构历史（哪些重构做过，效果如何）
├── 技术债务分布（哪些是刻意为之，哪些是遗留）
└── 代码组织的原则和惯例

第四步：文档输出
├── 架构演进文档（为什么变成了现在的样子）
├── 设计决策记录（ADR - Architecture Decision Records）
├── 核心流程图和数据流图
├── 关键约束和不变量清单
└── "如果你是新人，你需要知道的一切"文档
```

### 4.3 检查清单示例

**Code Review检查清单**：
```
代码质量：
├── □ 命名是否清晰表达意图？
├── □ 函数/方法是否短小且职责单一？
├── □ 是否有重复代码可以抽取？
├── □ 错误处理是否完备？
├── □ 边界条件是否处理？
└── □ 日志是否适度且有意义？

设计合理性：
├── □ 是否符合项目架构规范？
├── □ 接口设计是否合理？
├── □ 是否引入了不必要的依赖？
├── □ 是否考虑了扩展性？
└── □ 是否有过度设计？

安全性：
├── □ 输入是否做了校验和清洗？
├── □ 是否存在SQL注入/ XSS等风险？
├── □ 敏感数据是否加密处理？
├── □ 权限检查是否完备？
└── □ 是否有信息泄露风险？

性能：
├── □ 是否有N+1查询问题？
├── □ 循环内是否有不必要的操作？
├── □ 大数据量场景是否考虑分页/流式处理？
└── □ 缓存策略是否合理？

可维护性：
├── □ 是否有足够的注释（为什么，而非是什么）？
├── □ 测试是否覆盖关键路径？
├── □ 配置是否外置？
└── □ 是否方便后续扩展？
```

**故障排查检查清单**：
```
第一阶段：快速止血（15分钟内）
├── □ 确认故障现象和影响范围
├── □ 检查最近是否有变更发布
├── □ 检查监控指标是否异常
├── □ 检查日志中的错误信息
└── □ 评估是否需要回滚

第二阶段：定位根因（1小时内）
├── □ 缩小故障范围（网络/应用/数据库/缓存）
├── □ 检查相关系统是否正常
├── □ 分析流量和时间线
├── □ 排查资源使用情况（CPU/内存/磁盘/连接数）
└── □ 确认根因假设

第三阶段：修复验证
├── □ 实施修复方案
├── □ 验证修复效果
├── □ 确认无副作用
└── □ 更新监控告警

第四阶段：复盘总结（24小时内）
├── □ 编写事故报告
├── □ 记录时间线和决策
├── □ 提炼可复用的经验
├── □ 制定改进措施
└── □ 更新文档和检查清单
```

### 4.4 决策树示例

```
技术选型决策树：

选择数据库：
├── 数据是否有明确Schema？
│   ├── 是
│   │   ├── 读写比例？
│   │   │   ├── 读多写少 → MySQL/PostgreSQL
│   │   │   └── 写多读少 → 考虑Cassandra/ClickHouse
│   │   └── 是否需要复杂查询？
│   │       ├── 是 → PostgreSQL
│   │       └── 否 → MySQL
│   └── 否
│       ├── 是否需要高可用？
│       │   ├── 是 → MongoDB
│       │   └── 否 → Redis(缓存)/ES(搜索)
│       └── 数据量级？
│           ├── < 1TB → MongoDB
│           └── > 1TB → HBase/ClickHouse

选择消息队列：
├── 吞吐量要求？
│   ├── > 100万/秒 → Kafka
│   ├── 10万-100万 → RabbitMQ/RocketMQ
│   └── < 10万 → Redis Stream / SQS
├── 是否需要严格顺序？
│   ├── 是 → Kafka（Partition级别有序）
│   └── 否 → 任意MQ均可
└── 是否需要事务支持？
    ├── 是 → RocketMQ
    └── 否 → Kafka/RabbitMQ
```

```
# .cursorrules 示例

## 项目概述
这是一个基于 Next.js + TypeScript 的电商后台管理系统

## 技术栈
- Frontend: Next.js 14 (App Router), React 18, TypeScript 5
- Styling: Tailwind CSS, shadcn/ui
- State: Zustand
- API: tRPC
- Database: PostgreSQL + Prisma
- Auth: NextAuth.js

## 编码规范
- 使用函数式组件，优先使用 arrow function
- 自定义 Hook 以 use 前缀命名
- 组件文件名使用 PascalCase
- 工具函数文件名使用 camelCase
- 每个 API Route 添加输入验证 (zod)
- 优先使用 server components，按需使用 'use client'
- 错误处理使用 Result 模式，避免 try-catch 散落

## 架构约束
- 严格遵循分层：API Route → Service → Repository → Prisma
- 禁止在组件中直接调用 Prisma
- 所有 API 响应使用统一格式 { code, data, message }
- 路由按功能模块分组：/app/(admin)/products, /app/(admin)/orders

## 测试要求
- Service 层必须有单元测试
- 关键 API 必须有集成测试
- 使用 vitest + testing-library
```
