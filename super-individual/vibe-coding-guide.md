### 3.2 .cursorrules 最佳实践

```markdown
# .cursorrules 示例模板（Next.js SaaS项目）

## 项目概述
[一句话描述项目目标和核心价值]

## 技术栈
- Frontend: Next.js 14 (App Router), React 18, TypeScript 5
- Styling: Tailwind CSS, shadcn/ui
- State: Zustand
- API: tRPC / REST
- Database: PostgreSQL + Prisma
- Auth: NextAuth.js / Clerk
- Payment: Stripe
- Deploy: Vercel

## 编码规范
- 使用函数式组件，优先使用 arrow function
- 自定义 Hook 以 use 前缀命名
- 组件文件名使用 PascalCase
- 工具函数文件名使用 camelCase
- 每个 API Route 添加输入验证 (zod)
- 优先使用 server components，按需使用 'use client'
- 错误处理使用 Result 模式，避免 try-catch 散落

## 架构约束
- 严格遵循分层：API Route → Service → Repository → Prisma
- 禁止在组件中直接调用 Prisma
- 所有 API 响应使用统一格式 { code, data, message }
- 路由按功能模块分组：/app/(admin)/products, /app/(admin)/orders

## 性能要求
- 核心页面首屏加载 < 1.5s
- API响应时间 < 200ms
- 图片使用 Next.js Image 组件优化

## 安全要求
- 所有用户输入严格校验
- 敏感操作需要二次确认
- 遵循 OWASP Top 10 防护

## 测试要求
- Service 层必须有单元测试
- 关键 API 必须有集成测试
- 使用 vitest + testing-library
```

## 5. Vibe Coding最佳实践

### 5.1 效率最大化原则

| 原则 | 说明 | 效果 |
|------|------|------|
| **小步快跑** | 每次只让一个AI完成一个明确任务 | 减少错误，提高可控性 |
| **持续审查** | 不盲目接受AI生成的所有代码 | 保证代码质量 |
| **上下文管理** | 保持.cursorrules和CLAUDE.md更新 | AI理解更准确 |
| **版本控制** | 频繁commit，便于回滚 | 降低风险 |
| **测试驱动** | 让AI先生成测试，再写实现 | 提高可靠性 |
| **文档同步** | 代码变更后同步更新文档 | 降低维护成本 |

### 5.2 常见陷阱与避免

```
陷阱1：过度信任AI
├── 现象：AI生成代码直接部署，不审查
├── 后果：安全漏洞、性能问题、逻辑错误
├── 避免：始终审查关键代码，特别是安全和支付相关

陷阱2：上下文丢失
├── 现象：项目大了之后AI开始"遗忘"早期约束
├── 后果：代码风格不一致，架构偏离
├── 避免：定期更新.cursorrules，关键约束反复提醒

陷阱3：技术债务累积
├── 现象：快速迭代但从不重构
├── 后果：代码腐化，后期难以维护
├── 避免：每迭代3-5次安排一次重构对话

陷阱4：忽略边缘情况
├── 现象：AI只处理happy path
├── 后果：上线后大量Bug
├── 避免：主动要求AI处理错误情况和边界条件

陷阱5：安全盲点
├── 现象：AI生成的代码存在SQL注入、XSS等漏洞
├── 后果：数据泄露、被攻击
├── 避免：安全相关代码必须人工审查，使用安全扫描工具
```

## 7. 结语

Vibe Coding不是"不需要会编程"，而是**用更高层次的抽象来表达意图**。最优秀的Vibe Coder仍然是那些：

1. **理解底层原理**——知道AI生成的代码在做什么
2. **善于描述意图**——能清晰表达"要什么"而非"怎么做"
3. **保持批判思维**——不盲目接受，持续审查和优化
4. **拥抱快速迭代**——将省下的时间用于验证和优化产品

> **"未来的开发者不是写代码更快的人，而是能更清晰地描述问题的人。"**
